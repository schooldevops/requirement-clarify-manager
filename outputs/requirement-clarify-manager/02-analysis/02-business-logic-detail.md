# 비즈니스 로직 상세

## BL-REQ-001: 요구사항 입력(등록)
### 처리 흐름
1. 입력값 변수 수신 및 검증
   - 파일 포함 여부 확인
   - 필수 파라미터(JIRA 티켓명, 이름 등) 유무 확인
   - 버전 포맷(X.Y.Z) 검증
2. 파일이 포함된 경우
   - 파일 확장자가 `.md` 인지 체크 (`ERR-FILE-002`)
   - 파일 사이즈가 10MB 이하인지 체크 (`ERR-FILE-001`)
   - 파일을 Object Storage(S3 등)에 업로드하고 경로 확보
3. 요구사항 번호(JIRA 티켓명) 중복 여부 확인
   - 중복 시 `ERR-DUP-001` 발생
4. 연관 요구사항 번호 검증 (입력된 경우)
   - 존재하지 않을 경우 경고 로깅 (`ERR-REF-001`)
5. DB에 요구사항 데이터 저장 (상태: Draft)
6. 결과 반환

### 비즈니스 규칙
- **BR-REQ-001**: 요구사항 버전은 반드시 `^(\d+)\.(\d+)\.(\d+)$` 정규식을 만족해야 한다. (ERR-VAL-002)
- **BR-REQ-002**: 요구사항 데이터의 최대 크기 처리를 대비해 상세 텍스트는 CLOB/TEXT 형식으로 DB에 저장해야 한다.
- **BR-REQ-003**: 10MB 초과 파일은 어플리케이션 계층 도달 이전 또는 컨트롤러 단에서 빠르게 차단/거절한다.

### 예외 처리
| 예외 상황 | 에러 코드 | 처리 방식 |
|---------|---------|---------|
| 10MB 초과 파일 업로드 | `ERR-FILE-001` | HTTP 413 Payload Too Large 로 즉시 반환 |
| 미지원 확장자 파일 | `ERR-FILE-002` | HTTP 400 Bad Request 반환 |
| 필수 항목 누락 | `ERR-VAL-001` | HTTP 400 Bad Request 반환 |
| 잘못된 버전 포맷 | `ERR-VAL-002` | HTTP 400 Bad Request 반환 |
| 중복 요구사항 번호 | `ERR-DUP-001` | HTTP 409 Conflict 반환 |

## BL-REQ-002: 요구사항 상태 변경
### 처리 흐름
1. 대상 요구사항 ID 존재 여부 확인
2. 현재 상태에서 요청된 상태로의 전이 가능 여부 확인
3. 상태 전이 가능할 경우, 상태 업데이트 및 수정일시 갱신
4. 저장 및 결과 반환

### 비즈니스 규칙
- **BR-STS-001** (상태 워크플로우): `Draft` -> `Clarifying` -> `Clarified` -> `In Progress` -> `Done`
- **BR-STS-002**: 이전 단계나 동떨어진 단계로의 임의 점프는 불가하며, 정해진 순서 및 허용된 후퇴(예: Clarifying 으로 롤백)만 가능하다.

### 예외 처리
| 예외 상황 | 에러 코드 | 처리 방식 |
|---------|---------|---------|
| 비정상 상태 전이 시도 | `ERR-STS-001` | HTTP 400 Bad Request, 올바른 워크플로우 안내 |

## BL-REQ-003: 데이터 사전 자동 생성
### 처리 흐름
1. 대상 요구사항 텍스트 (또는 파일 내용) 로드
2. NLP/AI 엔진(자연어 처리 외부 시스템)에 텍스트 분석 요청
3. 형태소 기반 명사/동사 추출 완료 대기
4. 추출된 단어들을 기존 프로젝트 데이터 딕셔너리 테이블과 비교
5. 기존에 없는 단어는 신규 미정의(Undefine) 용어로 데이터 딕셔너리에 추가
6. 분석 완료 및 결과 알림 (비동기)

### 비즈니스 규칙
- **BR-EXT-001**: 분석은 대용량 텍스트 처리를 수반하므로 비동기로 실행되며, 즉시 202 Accepted 를 반환한다.
- **BR-EXT-002**: 데이터 사전은 반드시 "프로젝트 명" 기준으로 파티셔닝(또는 Tenant 분리)되어 겹치지 않게 관리되어야 한다.

### 예외 처리
| 예외 상황 | 에러 코드 | 처리 방식 |
|---------|---------|---------|
| NLP 엔진 타임아웃 | `ERR-EXT-001` | 재시도 3회 후 실패로 마킹, 알림 발송 |

## BL-REQ-004: 누락 용어 질의 피드백 (기획자와의 필드 단위 인터랙션)
### 처리 흐름
1. 미정의 용어 단건 조회 (GET)
   - 프로젝트 내 미정의(Undefine) 상태인 용어 중 하나를 기획자 화면에 제시
2. 단일 용어 명확화 제출 (POST)
   - 기획자가 제시된 단어에 대해 정의(DEFINE)를 수행(데이터타입, 길이, 설명 등 입력)하거나 건너뛰기(SKIP) 선택
3. 동시성 제어 (낙관적 락)
   - Version 칼럼을 사용해 다수 기획자가 동시에 동일 용어 수정 시 충돌을 방지하고, 늦게 요청한 사용자는 거부
4. 데이터베이스 갱신 및 상태 유지
   - DEFINE 시: 데이터 사전 값 갱신 및 정의 완료 처리
   - SKIP 시: 상태 유지하여 사이클이 한 바퀴 돈 이후 다시 제시되도록 처리
5. 다음 용어 즉시 반환
   - 완료 후 응답 페이로드에 '연속적인 다음 미정의 용어'를 함께 포함하여, 기획자가 화면 이탈 없이 계속 응답할 수 있는 피드백 루프 제공

### 비즈니스 규칙
- **BR-CON-001**: 동시에 여러 기획자가 동일 용어를 명확화 하려 할 때 낙관적 락킹 기반으로 충돌을 제어해야 한다. 늦게 요청한 사용자는 거부된다.
- **BR-REQ-004**: 미정의 용어가 0건이 될 때까지 지속적인 질의-응답(Q&A) 방식의 단건 루프를 보장해야 한다.

### 예외 처리
| 예외 상황 | 에러 코드 | 처리 방식 |
|---------|---------|---------|
| 동시성 수정 충돌 | `ERR-CON-001` | HTTP 409 Conflict 반환, 다른 담당자의 수정 사항 안내 및 다음 단어로 이동 권고 |
| 존재하지 않는 용어 | `ERR-NOT-FOUND` | HTTP 404 Not Found 반환 |

## BL-REQ-005: 요구사항 상세 분석 (Event Storming 파이프라인)
### 처리 흐름
1. 요구사항 ID 대상의 프로젝트 데이터 사전 확인
   - 프로젝트 데이터 사전에 '미정의' 항목이 남아있는지 체크
2. 미정의 항목이 존재한다면 `ERR-FLOW-001` 예외 발생 후 거부
3. 모두 정의되어 있을 경우 외부 AI 기반 분석 파이프라인에 텍스트+사전데이터 전송
4. 분석 엔진을 통해 Event Storming 각 컴포넌트(Actor, Event, Command 등) 식별
5. 식별 결과 DB 저장

### 비즈니스 규칙
- **BR-FLOW-001**: 분석 파이프라인은 데이터 사전이 100% 명확화 되거나 확인된 상태에서만 실행할 수 있다.

### 예외 처리
| 예외 상황 | 에러 코드 | 처리 방식 |
|---------|---------|---------|
| 사전 미완료 상태 분석 | `ERR-FLOW-001` | HTTP 422 Unprocessable Entity, 확인 필요 항목 안내 |

## BL-REQ-006: 시각화 데이터 조회 (통합 뷰)
### 처리 흐름
1. 요구사항 ID로 기본 메타데이터 조회
2. 연관 프로젝트의 최신 데이터 사전 조회
3. 요구사항 분석(Event Storming) 결과 객체화
4. 저장된 Mermaid 텍스트(Sequence, Usecase, BP 등) 로드
5. 하나의 Response DTO 로 통합하여 반환
