# 요건정의서: 요구사항 명세 관리 시스템 (MVP)

## 1. 개요
- 프로젝트명: Requirement Clarify Manager (요구사항 명세 관리 시스템)
- 요청 부서: 기획/사업팀
- 작성일: 2026-03-26
- 버전: v1.0

## 3. 비기능 요구사항
### NFR-001: 대용량 데이터 처리
- 응답 시간: 요구사항 상세에 수 MB에 달하는 텍스트가 입력되어도 저장 및 조회 시 3초 이내 응답을 보장해야 함.

### NFR-002: 데이터 격리성
- 데이터 격리: 데이터 사전 및 분석 결과는 반드시 "프로젝트 명" 기준으로 분리되어 조회가 이뤄져야 함.

## 4. 핵심 예외 케이스 (Edge Case) 10선
| EC-ID | 예외 상황 | 발생 조건 | 처리 방식 | 에러 코드 |
|---|---|---|---|---|
| EC-001 | 초대용량 파일 업로드 시도 | 10MB가 넘는 .md 파일 업로드 | 업로드 전 파일 사이즈 확인 후 거부, 10MB 제한 안내 | ERR-FILE-001 |
| EC-002 | 필수 입력값 누락 | 요구사항 번호나 이름 없이 저장 시도 | 저장 거부 및 필수항목 하이라이트 표시 | ERR-VAL-001 |
| EC-003 | 중복된 요구사항 번호 입력 | 이미 존재하는 요구사항 번호(Jira 티켓)로 신규 등록 시도 | 등록 거부, 기존 티켓 번호와 충돌 안내 혹은 리비전(버전업) 유도 | ERR-DUP-001 |
| EC-004 | 정의되지 않은 상태값 전이 | Draft에서 바로 In Progress로 강제 변경 시도 | 상태 전이 실패 처리, 올바른 워크플로우(Clarifying 등) 안내 | ERR-STS-001 |
| EC-005 | 명사/동사 추출 엔진 타임아웃 | 수 MB 텍스트 분석 중 NLP/AI 엔진 타임아웃 발생 | 비동기 처리로 전환하거나 부분 저장 후 실패 메시지 비동기 통지 | ERR-EXT-001 |
| EC-006 | 데이터 사전 정의 중복충돌 | 2명의 기획자가 동시에 동일 용어(미정의)를 정의 시도 | 낙관적 락(Optimistic Lock) 처리, 나중에 저장한 사용자에게 알림 | ERR-CON-001 |
| EC-007 | 비정상적인 버전 포맷 | Major-Minor-Patch(예: 1.0.0) 형식이 아닌 문자열(예: final_v1) 입력 | 정규식 검증 실패, 올바른 포맷(X.Y.Z) 안내 | ERR-VAL-002 |
| EC-008 | 연관 요구사항 번호 단절 | 존재하지 않는 요구사항 번호를 연관 번호로 입력 | 등록은 허용하되 경고 메시지 노출 (또는 엄격히 거부) | ERR-REF-001 |
| EC-009 | 데이터 사전 미완료 상태에서 분석 시도 | 용어 정의가 다 끝나지 않은 상태에서 Event Storming 분석 파이프라인 호출 | 호출 거부, 데이터 사전 완료 선행 안내 | ERR-FLOW-001 |
| EC-010 | 지원하지 않는 파일 포맷 업로드 | .md가 아닌 .doc, .pdf 파일 업로드 | 업로드 단계에서 확장자 체크로 필터링 | ERR-FILE-002 |

## 5. 제약사항
- 분석 파이프라인(Event Storming 등)은 데이터 사전 정의 단계 완료 후 수행되어야 함.
- 작성자는 MVP 버전에서 직접 문자열로 자신의 이름을 입력해야 함. (조직/사번 연동 시스템 없음)

## 6. 가정사항
- 기획자는 마크다운 포맷에 익숙하다.
- 요구사항 분석 외부 AI 파이프라인 엔진은 수 MB 텍스트를 처리할 성능적 여유가 있다.
