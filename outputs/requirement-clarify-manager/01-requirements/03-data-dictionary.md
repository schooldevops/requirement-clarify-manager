# 용어 정의서 (Data Dictionary)

`001-requirement-mvp.md` 상세 분석을 통해 추출된 전체 도메인 용어 데이터 사전입니다. 요구사항 내의 메타 정의를 반영하여 도출되었습니다.

## 1. 요구사항 메타데이터 (Requirement Metadata)

| 한글이름 | 영문이름 | 데이터 타입 | 데이터 길이 | 데이터 포맷 / 제약조건 | 동음이의어/이음동의어 | 설명 | 데이터 예시 | 비고 |
|---|---|---|---|---|---|---|---|---|
| 요구사항 번호 | Requirement ID | String | 50 | Unique Identifier | Jira 티켓 이름, 티켓 번호, 이슈 번호 | 요구사항 식별 고유 키. JIRA 티켓명과 동일해야 함 | `REQ-1234` | - |
| 연관 요구사항 번호 | Related Req ID | String | 50 | 등록된 Req ID 포맷 | 참조 요구사항, 관련 티켓 | 이전 또는 관련 요구사항과의 연결 고리 | `REQ-1000` | 없으면 빈 값 허용 |
| 요구사항 신규 구분 | Requirement Type | Enum | 10 | 신규/수정 | 신규여부, 수정구분 | 요구사항이 완전 신규인지 기존 기능의 수정인지 구분 | `신규`, `수정` | - |
| 프로젝트 명 | Project Name | String | 100 | - | 서비스 명, 시스템 명 | 요구사항 및 데이터 사전 격리의 기준이 되는 단위 | `상품관리 시스템` | 프로젝트별 데이터 격리 |
| 카테고리명 | Task Category | String | 50 | 사전에 정의된 값 한정 | 업무 카테고리 | 프로젝트 내의 업무 범주 분류 | `결제`, `회원` | 드롭다운 형태의 사전정의 |
| 요구사항 이름 | Requirement Name | String | 200 | Not Null | 요구사항 제목, 주제 | 요구사항에 대한 핵심 개요 문장/제목 | `장바구니 결제 로직 개선` | - |
| 작성자 | Author | String | 50 | Not Null | 작성자명, 작성자 사번 | 요구사항 작성자의 식별 정보 (이후 ID 연동 예정) | `홍길동`, `1111000` | MVP는 수기 입력 |
| 작성일 | Created At | DateTime | - | `YYYY-MM-DD HH:mm:ss` | 등록일, 생성일 | 요구사항이 처음 등록된 일시 | `2026-03-26 14:00:00` | 시스템 자동 입력 |
| 수정일 | Modified At | DateTime | - | `YYYY-MM-DD HH:mm:ss` | 갱신일, 변경일 | 요구사항이 마지막으로 수정된 일시 | `2026-03-26 15:00:00` | 시스템 자동 갱신 |
| 버전 | Version | String | 20 | `Major-Minor-Patch` 포맷 | 리비전 | 요구사항 문서의 리비전 상태를 M.m.p 규칙으로 표현 | `1.0.0`, `1.2.1` | 잘못된 포맷 거절 처리 |
| 요구사항 개요 | Req Overview | String | 4000 | Max 4000자 | 요약, 설명 | 요구사항 전반의 간략한 설명 | `사용자가 카트에 담긴...` | - |
| 요구사항 상세 | Req Detail | Text(CLOB) | 수 MB | - | 본문, 상세 구조 | 대량의 텍스트로 구성된 요구사항의 본문 내용 | (마크다운 텍스트) | NLP 파싱의 원천 데이터 |
| 요구사항 상태 | Req Status | Enum | 20 | `Draft`, `Clarifying`, `Clarified`, `In Progress`, `Done` | 진행 상태, 스탯 | 요구사항의 워크플로우 진행 단계 표시 | `Clarifying` | 순차/선행조건에 기반 전이 |

## 2. 데이터 사전 및 명확화 모듈 (Data Dictionary & Clarification)

| 한글이름 | 영문이름 | 데이터 타입 | 데이터 길이 | 데이터 포맷 / 제약조건 | 동음이의어/이음동의어 | 설명 | 데이터 예시 | 비고 |
|---|---|---|---|---|---|---|---|---|
| 데이터 사전 | Data Dictionary | Object | - | 프로젝트별 격리/종속 | 용어 집합, 단어장 | 요구사항에서 명사/동사로 자동 추출된 용어 모음 객체 | - | 시스템 주요 결과물 |
| 추출 용어 | Extracted Term | String | 100 | - | 어휘, 추출 단어 | 요구사항 상세 텍스트를 NLP 분석하여 도출된 명사/동사 | `상품`, `취소하다` | - |
| 피드백 질의 | Feedback Query | String | 500 | - | 명확화 요청, 질문 | 미정의된 용어에 대해 기획자에게 정의를 요구하는 텍스트 | `결제 대기 시간의...` | - |
| 용어 정의 상태 | Term Status | Enum | 20 | `Defined`, `Undefined`, `Skipped` | - | 사전 내 개별 용어의 정의 완료 여부 | `Skipped` | - |
| 낙관적 락 키 | Optimistic Lock | Number | - | 순차 증가 | 동시성 키 | 다수 기획자가 동시에 동일 용어 정의 시도 시 충돌 제어 | `1`, `2` | 버전업 충돌 제어 (내부용) |

## 3. 이벤트 스토밍 및 분석 산출물 (Event Storming & Views)

| 한글이름 | 영문이름 | 데이터 타입 | 데이터 길이 | 데이터 포맷 / 제약조건 | 동음이의어/이음동의어 | 설명 | 데이터 예시 | 비고 |
|---|---|---|---|---|---|---|---|---|
| 이벤트 스토밍 | Event Storming | Object | - | 분석 파이프라인 | 도메인 분석 기법 | 액터, 도메인 이벤트 등을 추출해 내는 일련의 과정 | - | - |
| 액터 | Actor | String | 50 | - | 사용자, 행위자 | 시스템을 사용하거나 이벤트를 촉발하는 주체 | `구매자`, `관리자` | - |
| 커맨드 | Command | String | 100 | 동사/행위 형태 | 명령 | 액터가 시스템에 지시하는 의도된 동작 | `결제를 취소한다` | - |

| 도메인 객체 | Domain Object | String | 100 | 명사 형태 | 엔티티, 모델 | 이벤트가 발생하고 상태를 관리하는 핵심 도메인 정보 | `결제`, `장바구니` | - |
| 애그리거트 | Aggregate | String | 100 | 명사 집합 | 트랜잭션 경계, 군집 | 도메인 객체들의 무결성을 보장하기 위한 묶음의 단위 | `주문/결제 애그리거트`| - |
| 외부 시스템 | External System | String | 100 | - | 서드파티, 연동 서버 | 시스템 바운더리 외부에 위치한 연동 대상 | `PG사`, `외부 알림톡` | - |
