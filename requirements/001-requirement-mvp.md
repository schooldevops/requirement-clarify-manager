# 001-requirement

- 이 요구사항은 "요구사항 명세 관리 시스템" 의 MVP 기능을 구현하기 위한 요구사항이다. 

## 요건 상세

- 요구사항 입력은 2가지 방식이 존재한다.
    - 파일 등록 방식: .md 파일로 된 기획자의 요구사항을 파일로 입력 받는다.
    - 사용자 직접 입력 방식: 요구사항 정의 페이지에서 기획자가 요구사항을 입력한다. 
- 요구사항 입력은 다음 내용을 담고 있다.
    - 요구사항 번호: jira 티켓 이름과 동일하며, 고유한 값이다. 
    - 연관 요구사항 번호: 이전 요구사항과 연관이 있는 경우 입력한다. 
    - 요구사항 신규 구분: 요구사항이 신규인지, 수정인지 등을 입력한다.
    - 요구사항 프로젝트 명: 요구사항에 해당하는 프로젝트 명에 들어간다.
    - 요구사항 업무 카테고리명: 요구사항 카테고리는 사전에 정의된 업무 
    - 요구사항 이름: 요구사항에 대한 간단한 이름을 작성한다.
    - 작성자: 초기 MVP에서는 사용자가 직접 자신의 이름이나 사번을 입력한다. 향후 추가 기능 개발에서 자동으로 사용자의 아이디를 바탕으로 이름이 입력이 될 것이다. 
    - 작성일: 요구사항을 작성한 일자
    - 수정일: 요구사항을 수정한 일자
    - 버전: 요구사항의 버전을 나타내며 일반적은 Major-Minor-Patch 형식으로 구성된다. 
    - 요구사항 개요: 요구사항에 대한 개략적인 개요를 입력하며, 약 4000자 정도의 텍스트를 입력할 수 있다. 
    - 요구사항 상세: 요구사항 상세는 매우 큰 텍스트로 구성될 수 있으며 수 mb 정도의 텍스트도 입력이 될 수 있다. 
    - 요구사항 상태: 요구사항의 상태를 나타내며, 다음 5가지 상태가 존재한다.
        - Draft: 요구사항이 작성되었으나, 아직 검토되지 않은 상태
        - Clarifying: 요구사항이 검토중이며, 명확화가 필요한 상태
        - Clarified: 요구사항이 명확화되었으며, 개발이 가능한 상태
        - In Progress: 요구사항이 개발중인 상태
        - Done: 요구사항이 완료된 상태
- 요구사항을 입력하고 등록하면 시스템은 요구사항의 내용에서 명사, 동사 등을 추출하여 데이터 사전을 만든다. 
- 데이터 사전은 "요구사항 프로젝트 명" 을 기준으로 관리 되어야하며, 동음이의어, 이음동의어, 한글이름, 영문이름, 데이터 타입, 데이터 길이, 데이터 포맷, 데이터 제약조건, 데이터 예시, 데이터 설명, 데이터 비고 등을 포함한다.
- 데이터 사전이 우선 완성이 되면, 요구사항 명세 관리 시스템은 데이터사전 정의부터 기존 저장되어 있는 데이터 사전을 바탕으로 정의가 되어 있지 않은 부분에 대해서, 기획자에게 역으로 질문하여 채워나가는 형태임
- 기획자가 정의하지 못할경우, skip하고 다음번에 추가로 작성이 가능함
- 데이터 딕셔너리가 완료되면 다음 단계로 요구사항 상세 내용을 분석한다. 
- 요구사항 상세 분석에서는 요구사항 상세 내용을 이용하여 요구사항 명세 관리 시스템은 Event Storming 형식으로 액터, 도메인 이벤트, 커맨드, 도메인 객체, aggregate, 외부 시스템 등을 추출한다. 
- 최종적으로 요구사항 분석 결과로 다음 형식의 뷰가 만들어 저여한다.
    - 데이터 딕셔너리
    - 요구사항 명세 원본 파일 (.md) 파일로 구성
    - Event Storming 결과물
    - 비즈니스 프로세스 다이어그램
    - 시퀀스 다이어그램
    - 유스케이스 다이어그램
- 최종 결과물은 향후 개발시에 활용하고, 추가 요청건이 들어오는 경우 이전 요구사항과 연관이 있는지 등을 통해서 신규, 수정 등을 판단 할 수 있어야한다.
